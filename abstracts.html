RubyMotion - Developer happiness for iOS
Many of us want to be able to write apps for the iOS platform, but the thought of having to learn the full Xcode stack and tons of Apple SDKs seems quite daunting. There are alternatives like PhoneGap etc., but they just don't provide the same experience as something fully native except for the simplest of apps (no matter what the HTML5-pundits say). Being a dialect of Ruby, it brings the same developer happiness it's famous for on the desktop to the iOS platform. In this lightning talk I'll show you how easy it is to get started.

Integrasjon som skalerer

Responsive design is the answer - but what was the question?
In the SF book "The Hitchhikers Guide to the Galaxy", one of the main characters views the universe through a pair of Peril Sensitive sunglasses - sunglasses that blocks out anything potentially dangerous in your field of vision so you can carry on doing what you're doing without worrying too much.
The sunglasses operate by projecting directed "SEP-fields" (where SEP is short for "Somebody Else's Problem") at the wearer in order to trick the wearer's brain into ignoring any potential hazards and thereby simply stopping the wearer from seeing them.

This session is about exploring the current state of Responsive web design (RWD) from selected angles to find out if we somehow unintentionally haven't ended up being influenced by something like a collective SEP-field where essential aspects of web standards, accessibility, Ux etc. might not be optimally handled (Because they’re somebody else’s problem) in our rush to cater to our customers cravings for responsive design solutions…
… and if that should be the case, to inspire to further discussions on how we can make our best practices for delivering good user experiences on the web even better.

· Responsive - to what?
· RWD vs. Separate sites/channels – what’s the rumpus?
· About Ux and providing choice when needed
· Applying TWAP (as in "The Web Ain't print") principles to RWD
· Configurability options within the current design - where did those go?
· What would User adaptible RWD look like?

· If you structure content – your design will respond
· WYSIWYG is dead - long live WYSIWYG
· Structured content – the new crown Prince?
· The Markdown effect
· A11y / SEO / SEC aspects of structured content

· The open web stack to the rescue!
· HTML5 - an umbrella term for much more than just markup
· Scripting dependent web design - dream or reality ?
· Can we start rethinking established design patterns based on the open web stack concepts (like scripting capabilities) now?
· Should we?
· Arent' we already doing it?
· … and where can we go from here?

Hvordan strukturere og skrive modulær JavaScript

I motsetning til mange andre programmeringsspråk har ikke JavaScript eksplisitt støtte for å definere moduler og avhengigheter mellom frittstående deler. Nettopp dette er et vanlig argument blant skeptikere, når JavaScript kodebaser vokser blir det vanskeligere og vanskeligere å strukturere koden.

Er slike utsagn mangel på forståelse for språket og økosystemet rundt, eller har skeptikerne et gyldig argument?

Vi skal i denne presentasjonen se på noen måter å strukturere kode i "vanilla" JavaScript. Etter dette kommer vi til å ta for oss to modulformater, AMD og CommonJS, som begge forsøker å formalisere modularisering av JavaScriptkode. Hva er forskjellen mellom de to? Er det noe hensikt å ta de i bruk på mindre prosjekter, og hvilke verktøy finnes det der ute som støtter opp under disse? Til slutt kommer vi til å se på hva vi kan forvente oss på dette området av neste JavaScript versjon.

Dette er en presentasjon for deg som er nybegynner i JavaScript men ønsker å lære mer om organisering av avhengigheter og strukturering av kodebase i JavaScript.

Maintainable Ruby on Rails
Since Rails 3.2, Engines has become the standard way of splitting application functionality into smaller isolated pieces. A Rails Engine can contain the same functionality as a normal Rails application, with the added benefit of being distributable as a gem which can be plugged into your applications. This is key for both creating reusable components and separating concerns within larger applications.

Workshop Goal:
Get familiar with using and creating Rails Engines, and understand how they can help improve the maintainability aspects of your apps. We’ll create a small application which will demonstrate how they work, starting by using popular 3rd party engines, before creating our own engine. We’ll talk about what kind of functionality fits best within an engine, and provide general advice on best practices.

Agenda:
- What’s a rails engine and why would you want to use them?
- Best practices for creating engines
- Using 3rd party engines within your application
- Creating your own engine
- Wiring it all together
- Final thoughts / recommendations

Participant requirements: Laptop with at least Ruby 1.9.3 (or greater) installed.

Duration: 3 hours.
